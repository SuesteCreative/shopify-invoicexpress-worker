Integração Cliente Novo — Shopify → InvoiceXpress Auto-Webhooks

Fluxo novo: OAuth offline → access token → instalação automática de webhooks via API. Não requer criar webhooks manualmente no Shopify Admin.

Aviso: Não partilhar Client Secret, tokens ou authorization codes por chat, email ou screenshots. Tratar como credenciais.
Diferenças face ao fluxo manual antigo:
  • Os 4 webhooks são instalados via POST /admin/api/.../webhooks.json — não no Settings → Notifications.
  • Não é necessário copiar o "Webhook Signing Secret" do Shopify Admin. Webhooks criados via API usam o Client Secret da app (já guardado no worker como SHOPIFY_API_SECRET).
  • Custom Apps (uma loja só) têm acesso automático a Protected Customer Data — não há formulário a preencher.

Parte A — Criar App no Shopify Dev Dashboard

Aceder a dev.shopify.com/dashboard (ou partners.shopify.com conforme o caso) → Apps → Create app → manualmente.

Embed app in Shopify adminDESLIGADO
Allowed redirection URL(s)https://example.com/
Scopes (Admin API)read_customers, read_discounts, read_order_edits, read_orders, read_all_orders, read_products
Nota: read_webhooks / write_webhooks já não existem como scopes válidos. A criação de webhooks via Admin API é autorizada pelo scope do tópico (ex: orders/* requer read_orders). Como Custom App, o acesso a Protected Customer Data é automático — não há formulário PCD a preencher.

Versão e Release

  1. Sempre que alterares scopes ou redirect URLs, criar nova versão e fazer Release.
  2. Versão tem de estar Active / Ativa.
  3. Após release, copiar (em Definições / Settings da app):
    • Client ID (API key)
    • Client Secret (também serve para verificar a assinatura dos webhooks instalados via API)
    • Domínio nativo da loja (ex: quickstart-66f9e5ef.myshopify.com)
Não fazer instalação direta da app pelo botão da lista. Usar o Builder da Parte B para construir o link /authorize.

Parte D — Instalar Webhooks via API Auto

Em vez de criar 4 webhooks manualmente no Shopify Admin, fazemos 4 chamadas POST ao endpoint /admin/api/2026-01/webhooks.json. O token só precisa do scope do tópico (read_orders cobre orders/* e refunds/create).

Resposta esperada por chamada: 
HTTP/2 201
{ "webhook": { "id": 9876543210, "topic": "orders/paid", "address": "https://.../webhooks/shopify/orders-paid", "format": "json", ... } }
Códigos:
  • 201 Created → instalado.
  • 422 Unprocessable Entity com "address has already been taken" → já existia, OK.
  • 403 Forbidden → token sem scope do tópico OU app não distribuída como custom. Repetir Parte A com scopes correctos.
  • 401 Unauthorized → token inválido. Repetir Parte B+C.
Atenção à assinatura dos webhooks: webhooks criados via API são assinados com o Client Secret da app (shpss_…), não com o secret de "Your webhooks will be signed with" do Shopify Admin (esse só serve para webhooks criados manualmente em Settings → Notifications). No worker, o HMAC tem de ser verificado contra a env var SHOPIFY_API_SECRET (Client Secret).

Parte G — Troubleshooting

Sem code no redirect

  • Redirect URI aponta para servidor que força login → usar https://example.com/.
  • Embed app in Shopify admin activo consome o code → desligar.
  • Redirect URI tem de ser exactamente o que está em Allowed redirection URL(s), incluindo barra final.

Erro ao trocar code por token

  • code expirado → repetir Parte B para obter novo.
  • code já usado → cada code só pode ser trocado uma vez.
  • Verificar que client_id, client_secret e shop correspondem à mesma app/loja.

Scope read_webhooks / write_webhooks marcado como inválido

  • Esperado. Esses scopes foram removidos. Usar apenas os 5 da tabela da Parte A.
  • Webhook subscription via API só requer o scope do tópico (read_orders para orders/*).

POST /webhooks.json devolve 403

  • Confirmar que o token tem read_orders (no scope da resposta da Parte C).
  • Confirmar que a versão da app está Active com os scopes correctos.
  • Se a app for Public (não Custom), pode faltar aprovação de Protected Customer Data — manter app como custom (distribuição directa por OAuth a uma loja).

POST /webhooks.json devolve 422 "address has already been taken"

  • O webhook já existe (de tentativa anterior ou criação manual). Tratar como sucesso, ou apagar antes: DELETE /admin/api/2026-01/webhooks/{id}.json.

Webhooks chegam ao worker mas falham HMAC

  • Webhooks criados via API são assinados com Client Secret, não com o "Your webhooks will be signed with" do Admin.
  • Confirmar que SHOPIFY_API_SECRET (env do worker) = Client Secret da app.
  • Header é X-Shopify-Hmac-SHA256. Comparar HMAC base64 do raw body com esse header.

Webhooks não disparam

  • Listar com curl da Parte D → confirmar que os 4 estão lá.
  • Confirmar Format = json e Address sem trailing slash extra.
  • Encomenda de teste tem de ser paga (não só criada) para disparar orders/paid.
